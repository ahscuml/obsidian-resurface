## 三件被独立证据反复印证的事（最高信心）

这三件事在 4 份报告中都出现，且都有元分析或大样本支撑，是插件的**不可动摇的基础**。

### 1. 检索练习 >> 被动重读

| 研究 | 条件 | 延迟后结果 |
|---|---|---|
| Roediger & Karpicke 2006 | 四次重读 vs 一次测试 | 1 周：40% vs 56% |
| Karpicke & Blunt 2011 (Science) | 概念图 vs 检索练习 | 1 周：45% vs 67%，d ≈ 1.07 |
| Larsen 2009 | 重读 vs 测试 + 反馈（医学住院医师真实教学） | **6 个月：26% vs 39%，Cohen's d = 0.91** |
| Adesope 2017 元分析 | ~150 对比较 | Hedges' g ≈ 0.50–0.61 |
| Dunlosky 2013 | 10 种学习技术评估 | "Practice testing" 为仅有的两种"高效用"之一 |

**题型效应量排序**（Rowland 2014 元分析）：
- 自由回忆 g = 0.61
- 提示性回忆 g = 0.61
- 再认/多选 g = 0.29
- **越需要"自己生产答案"，效应越大**

👉 **产品含义**：**纯推送重读方案是伪科学**。即使要做"轻量模式"，也必须注入生成/检索成分（哪怕只是"先显示标题让用户想 5 秒"）。

### 2. 间隔 >> 集中

| 研究 | 发现 |
|---|---|
| Cepeda 2006 元分析（317 个实验、271 个对比） | **259/271 个支持间隔**；1 年以上保留，间隔比集中提升 3 倍 |
| Bahrick 1993（9 年外语词汇追踪） | 56 天间隔 vs 14 天间隔，长期保留翻倍 |
| Dunlosky 2013 | "Distributed practice" 为仅有的两种"高效用"之一 |
| Kerfoot 2007（泌尿外科 RCT, n=537） | 间隔 + 检索 vs 一次性：长期保留 15.2% vs 3.4%（~4× 效率） |

👉 **产品含义**：这就是插件存在的理由。

### 3. 间隔效应的量化规律

**Cepeda 2008 "Temporal Ridgeline"** 给出可落地公式：**最优复习间隔 ≈ 目标保留期 × 10-20%**，且**比例随目标期延长递减**：

| 目标记忆期 | 最优 gap | gap/RI |
|---|---|---|
| 7 天 | ~1 天 | ~15% |
| 35 天 | ~11 天 | ~30% |
| 70 天 | ~14 天 | ~20% |
| 350 天 | ~21 天 | ~6% |

**FSRS 公式** 独立地以 DSR 模型数学化了这个直觉：
$$SInc \propto (e^{w_{10}(1-R)} - 1)$$

也就是说——**越接近遗忘时成功回忆，稳定度涨得越多**。

👉 **产品含义**：**首次复习不应该是第二天**，应设为 2-5 天后。让用户"努力回忆"比"舒服回忆"对长期记忆更好。

## 算法选型：几乎是定局

| 算法 | srs-benchmark Log Loss | 结论 |
|---|---|---|
| SM-2 | ~0.49 | 不推荐作为默认 |
| Duolingo HLR | 0.47 | 被 FSRS 显著击败 |
| **FSRS-6** | **0.346** | **默认算法** |
| SM-17（闭源） | 0.432（独立基准） | FSRS 也击败它 |
| RWKV-P（神经网络 2.7M 参数） | 0.277 | 上限参考，不实用 |

**关键发现**：
- `ts-fsrs`（MIT 许可、零依赖、原生 TypeScript）可直接在 Obsidian 插件里用
- FSRS 冷启动有官方默认参数，新用户第一天就能用
- 本地 CPU 上 10 万条复习记录 optimize 仅 < 10 秒，不需要云端
- 在 `fsrs-vs-sm17` 的 SuperMemo 用户数据独立基准上，FSRS-6 也击败 SM-17（0.368 vs 0.432）—— 闭源优势站不住

不需要走"先 SM-2 后切 FSRS"的过渡路径。FSRS 默认参数在任何数据量下都不差于 SM-2。

## 两个容易踩的坑（被证据反复警告）

### 坑 1：JOL（学习判断）偏差

**Koriat & Bjork, Kornell & Bjork** 系列研究：学习者系统性地**高估被动重读、低估检索练习**。Karpicke & Roediger 2008 发现：即使学生亲身经历过"测试组成绩更高"，下一次仍然倾向于选择重读。

👉 **产品含义**：
- **不能把"用户感觉舒适"当成优化目标**
- "先想 5 秒再翻面"这类设计会让用户不爽但对记忆更好
- UI 文案要直面这个张力
- 一个有证据支撑的 UI 功能：**展示"你上次自评能记住 X%，实际回忆出 Y%"的校准反馈**，数次循环后能显著改善元认知准确性

### 坑 2："扩展间隔"的神话

**Karpicke & Roediger 2007/2010**：只要首次复习足够延迟，**等间隔 ≈ 扩展间隔**。SM-2 的 EF 递增不是"扩展"起作用，而是"让首次检索够难"。

👉 **产品含义**：不必迷信复杂的递增曲线。FSRS 公式形式是对的（基于 DSR 而非"扩展"），这是它设计上更干净的一个原因。

## 几个有分歧或有边界的问题

### 分歧 1：Interleaving 要不要无脑做？

- **支持**：Rohrer & Taylor 2007 小学数学实验——长期效果 77% vs 38%
- **边界**：Carvalho & Goldstone 等人——当类别相似度低、或学习者尚未建立基础理解时，**block 反而更优**

👉 **产品含义**：**不能做全局随机打散**。应做"新学阶段 block 一轮，熟练后 interleave"的分层调度。

### 分歧 2：检索练习对"复杂概念/程序性知识"效果会打折

- **Van Gog & Sweller 2015**：高元素交互性材料，RP 优势缩小或消失
- **Kang 2016**：间隔效应在复杂概念和程序性知识上衰减
- **Karpicke 2015** 回应：多数是实验设计缺陷
- **Kerfoot 系列**：对标准化考试的高阶推理迁移**没有清晰效果**

👉 **产品含义**：**你要复习的知识是什么形态？** 事实/词汇最稳；概念/程序性需要搭配 elaboration 或 worked example。这不是"一刀切调度"能解决的。

### 分歧 3：扩展/可选增强的 elaboration 要不要做

- **Dunlosky 2013** 把 Elaborative Interrogation 和 Self-Explanation 列为"中等效用"
- 在 Obsidian 生态里，**双链本身就天然提供了 elaboration 的 cue**——这是"别家做不到"的独特机会

## 被证据明确否定的做法（不要做）

| 做法 | 来源 | 原因 |
|---|---|---|
| Progressive Summarization（多轮高亮） | Dunlosky 2013 | 本质是低效用的 highlighting + summarization 叠加，无实证 |
| Cornell 三栏模板强推 | Jacobs 2013 等 | 多数研究未见显著学习效果，只是笔记结构 |
| Feynman 技法品牌功能 | - | 网上流传的 fMRI/Science 数据是 AI 内容农场**伪造**；底层机制由 self-explanation 覆盖 |
| 多选题 + 诱惑干扰项 + 无反馈 | Roediger & Marsh 2005 | 会产生"负面检索"强化错误 |
| 展示"今天会忘 62%"这类数字 | - | Ebbinghaus 曲线是无意义音节实验结果，在笔记上不成立，是伪精确 |

## Obsidian 生态的独特机会

这是别家（Anki、RemNote、Readwise）做不到或做得更笨的地方：

1. **双链 → 自然 interleaving 与 elaboration**
   - 复习某卡时，右栏自动显示其**反向链接**与**同一 tag 的其他卡**
   - 既实现 elaborative interrogation 的提示，又为 interleaving 提供语义化的混排依据

2. **atomic note = atomic card**
   - 不要在笔记外另建"卡片库"
   - 让每张原子笔记内嵌 `?::` 或 `^card` 语法，自动成为卡片
   - Zettelkasten 工作流与复习流合并，避免 Anki 用户常见的"两个系统割裂"

3. **Dataview 驱动可配置的复习队列**
   - 用户用 Dataview 查询语句自定义今日队列
   - 把 interleaving 策略交给用户按学科特性调整

4. **笔记修改时间驱动的 re-trigger**
   - 当某原子笔记被编辑或在其他笔记中被新链接时，自动把关联卡片重新推进队列
   - 相当于 Ahrens 的"笔记生长"触发了自然的 distributed practice

5. **Canvas / Graph 作为 retrieval cue**
   - 复习后让用户把卡拖到某个主题簇里，等于强制一次 elaboration + spatial encoding
   - 文本-only 的工具做不到

## 基于证据的插件核心架构（初稿）

4 份证据收敛成一个相当清晰的产品形态：

```
┌─ 输入层：笔记选择（哪些笔记进入复习池）
│     待决策：frontmatter / tag / 文件夹 / 全量
│
├─ 卡片层：一笔记多卡，atomic note = atomic card
│     · cloze 遮盖（证据最强，形态最轻）
│     · 笔记标题 → 内容回忆（Obsidian 最自然）
│     · 用户可选的简答回忆（效果最好，门槛最高）
│
├─ 调度层：FSRS-6（ts-fsrs）
│     · 冷启动默认参数
│     · ≥1000 次复习后 auto-optimize
│     · 首次复习 2-5 天后（而非当天）
│     · 分层 interleave（主题/tag 聚类打散，但保留新卡 block 选项）
│     · 暴露 desired_retention 参数（默认 0.9）
│
├─ 交互层："先答后看"为默认
│     · 评分：2 档（Again/Good）或 4 档（Anki 标准）
│     · 必须显示原文反馈（完整答案，非 ✓/✗）
│     · 反馈可轻微延迟 1-2 秒
│     · 可选的难卡触发：elaboration 提示
│     · 可选的 JOL 校准：显示"自评 X% vs 实际 Y%"
│
└─ 数据层：完整 revlog 从 Day 1 就存
      · schema: card_id, timestamp, rating, elapsed_days, state, outcome
      · 为将来的 optimizer 训练和算法切换留出空间
```

## 待决策的设计问题（基于证据仍需用户偏好才能定）

1. **内容形态** — 事实 / 概念 / 程序性？影响"默认卡片类型"和"算法是否够用"。
2. **复习触发：主动 vs 被动** — 推送提醒 vs 用户主动打开？
3. **笔记如何纳入复习池** — frontmatter / tag / 文件夹 / 全量（排除式）？
4. **卡片生成方式** — 手动标记 cloze / LLM 自动生成 / 笔记标题自动成卡 / 混合？
5. **UI 如何处理 JOL 偏差** — 多强地教育用户？
