> 最后更新：2026-04-21
> 关联调研：见 [../research/](../research/)

## 一、产品定位

### 一句话描述

**一款让你在 Obsidian 里写过的笔记，按科学的节奏重新找到你的复习插件。**

### 核心哲学

1. **笔记 = 复习对象**。不在笔记之外建"卡片库"，不要求用户额外制卡。笔记本身就是被复习的最小单元。
2. **复习 = 重新见笔记**。不是考试，不是检测，而是让一条笔记在合适的时刻重新进入你的注意力。
3. **不打扰**。不推送、不催促、不替用户做决定。用户回到 Obsidian 时才看到复习提示。
4. **复习现场完成所有决策**。不需要提前制卡、提前打标签、提前做管理。管理是复习的副产物。
5. **复习时可顺手编辑笔记**（"二次复活" 的世界观）。笔记被看到时是活的——可以改、可以加链接、可以生长。

### 与 Anki 的本质差异

| | Anki | Resurface |
|---|---|---|
| 复习单元 | 人造卡片（正/反面） | 笔记本身 |
| 制卡工作量 | 需要专门造卡 | 零，笔记写完即入池 |
| 复习时能做什么 | 看答案 + 打分 | 看笔记 + 打分 + **可编辑** |
| 管理方式 | 提前分 deck、配置 | 复习现场决定排除 |

## 二、用户目标

- **主要目标**：对抗遗忘，让自己能记住笔记内容
- **次要目标**：在复习过程中加深理解、建立笔记之间的联系

### 典型笔记形态

- 原子化短笔记为主（Zettelkasten 风格）
- 大部分是文字笔记
- 标题通常已经是问题或概念名，天然是 cue

## 三、核心算法

### 选型：FSRS-6

- **实现**：`ts-fsrs`（MIT 许可，TypeScript，零依赖）
- **冷启动**：使用 FSRS 官方默认参数，无需训练即可工作
- **优化**：攒够 ≥ 1000 次复习后，支持用户手动触发本地 optimizer 重新 fit 参数
- **理论基础**：DSR 模型（Difficulty / Stability / Retrievability）
- **为什么不是 SM-2**：srs-benchmark 实测，FSRS Log Loss 相对 SM-2 降低约 30%
- **证据来源**：见 [../research/02-SRS算法对比-FSRS深度剖析.md](../research/02-SRS算法对比-FSRS深度剖析.md)

### 关键参数

- **desired retention**：默认 0.9（用户可在高级设置中调整 0.80-0.97）
- **首次复习间隔**：3 天 ± 1 天（可配置）
- **评分档数**：默认 2 档（Again / Good），可切 4 档（Anki 标准）

## 四、笔记入池与排除

### 入池规则

- **默认所有笔记自动入池**，零配置
- 笔记创建后经过首次复习间隔（3 天 ± 1 天）开始调度

### 排除机制

- **只能在复习现场排除**（非复习时段不需要配置）
- 排除后笔记永久从复习池移除，但笔记文件本身零污染
- 存储位置：Obsidian 插件的 data.json（不修改笔记内容）
- 排除记录可在高级设置中查看和恢复

### 待决策（问题 B，延后讨论）

- 笔记稳定期：新建笔记是否立即入池？还是留几天"编辑期"？
- 空笔记、草稿、临时笔记的处理
- 智能默认排除（Daily Notes、Templates 等典型目录）

## 五、复习触发机制

### 方式：被动视觉提示（不推送）

- **侧边栏 ribbon badge**：显示今日待复习数字，一直可见
- **启动时顶部通知**：进入 Obsidian 时短暂弹出（3-5 秒自动消失）
  - 有待复习时：显示"今天有 N 条笔记想重新见你"
  - 无待复习时：**不显示**（避免打开即被通知的疲劳）

### 不做的事

- ❌ 系统通知（macOS/iOS notification）
- ❌ 强制的"每日复习"弹窗
- ❌ 定时器/闹钟
- ❌ 邮件提醒

## 六、每日上限与逾期堆积

### 每日上限

- **默认值**：15 条（用户可调 5-50）
- **推算依据**：一条笔记停留 1-3 分钟 × 15 条 ≈ 20-45 分钟/天

### 今日列表的计算时机

**实时计算**：每次用户打开复习界面（侧栏打开 / 点下一条）时，扫一遍 data.json，按 R 值排序取前 N 条作为"今日待复习列表"。

- **不做日缓存**。性能上 5000 条笔记级别扫一遍是毫秒级，无需优化
- 这样能让"笔记编辑后 Stability 降低 → 该笔记 R 值下降 → 应立即进入复习列表"这个链路**立即生效**
- 日边界（凌晨 04:00）的作用：决定哪些笔记"今日已复习"被排除、streak 是否延续，不用于冻结列表

### 日边界

- **凌晨 04:00** 作为"新一天"的分界线（参考 Anki 默认）
- 照顾夜猫子：凌晨 1-3 点复习仍然算昨天
- 已复习笔记在到 04:00 之前不会再次出现

### 超量笔记的处理

**策略：什么都不做，让 FSRS R 值自动消化**

- 超出当日上限的笔记 FSRS 状态保持不变
- 次日和后续几天里，因 R 值持续下降，这些笔记会被"最急排序"自动优先推出
- **不需要任何额外的"逾期处理"逻辑**

### 用户多天不打开后的场景

- 不做特殊提示
- 打开时同样按 R 值排序
- 最"快要忘"的笔记被优先展示，自然消化堆积

## 七、产品形态

### 结构

```
┌─────────────── Obsidian 主窗口 ───────────────────┐
│                                                    │
│  主工作区：                    │  侧边栏（右侧）:   │
│  ─────────────                │  ──────────      │
│  用户原有的笔记 tab 保持不变    │  🌱 复习面板      │
│                                │                   │
│  + 一个"复习专用 tab"          │  - 今日进度       │
│    （复习期间所有笔记在此流转） │  - 当前笔记 cue   │
│                                │  - 展开/评分按钮  │
│                                │  - 统计 & 鼓励    │
└────────────────────────────────────────────────────┘
```

### 实现方式

- 侧边栏面板：Obsidian `ItemView`，默认打开在右侧（`getRightLeaf`）
- 允许用户拖动到 tab 区或左侧栏（Obsidian 原生能力）
- "复习专用 tab"：**按需创建**——用户点"展开正文"时如果 tab 不存在就创建，如果已存在就复用。用户关掉了就关掉了，下次点"展开"再新建

### 无会话状态设计

复习不是一个有"开始—进行—结束"状态的模式。它就是对"今日待复习列表"的反复访问：

- 关掉侧栏 / 关掉复习 tab / 关掉 Obsidian，都不算"中断"
- 同一天内再次打开，从"列表里第一个未复习的"继续
- 过了日边界（凌晨 04:00），列表自然重算，不存在"恢复到昨天哪一条"
- 侧栏和主区两个 tab 物理上独立——进入下一条 ≠ 影响用户其他 tab 的编辑

## 八、复习流程（完整时间线）

### 阶段 0：闲置状态

用户在 Obsidian 里正常工作，主区是用户自己的笔记 tab。
侧边栏 ribbon 上显示一个小数字（今日待复习数）。

### 阶段 1：启动复习

用户点击 ribbon badge → 侧边栏展开：

```
┌─────────────────────┐
│ 🌱 1/12 今日待复习   │
│                     │
│ ### 笔记标题        │
│                     │
│ > TLDR: 摘要...     │
│                     │
│ ┌─────────────────┐ │
│ │  展开正文       │ │
│ └─────────────────┘ │
│                     │
│ [ 不再复习 ]        │
└─────────────────────┘
```

**关键**：主区不动，用户原状态完整保留。

### 阶段 2：看正文

用户点"展开正文" → 主区打开"复习专用 tab"，显示完整笔记：

```
侧边栏变为：
┌─────────────────────┐
│ 🌱 1/12              │
│                     │
│ ### 笔记标题        │
│                     │
│ [ Again ]  [ Good ] │
│                     │
│ [ 不再复习 ]        │
│ [ 下一条 → ]        │
└─────────────────────┘
```

### 阶段 3：评分

用户点 Good（或 Again）→ FSRS 调度下次时间。

**主区停留在当前笔记**，用户可自由编辑、加链接、写新想法。

### 阶段 4：进入下一条

用户主动点"下一条 →" → 复习专用 tab 切换到下一条笔记内容，侧边栏回到阶段 1 样式但数字 2/12。

**可选**：设置中开启"评分后自动进入下一条"，阶段 3 直接进入阶段 4。

### 阶段 5：全部完成

```
┌─────────────────────┐
│     🌱              │
│                     │
│  今日的 12 条笔记    │
│  都在你这里重新      │
│  活了一次            │
│                     │
│  ─────────────      │
│                     │
│  连续 7 天陪伴笔记   │
│  累计复习 435 次     │
│  下一批约在明天      │
└─────────────────────┘
```

### 关键节奏特性

1. **主区和侧边栏状态严格对应**：侧边栏显示 X 时，主区要么是 X，要么是用户原工作（cue 阶段）
2. **复习专用 tab 单一**：不会因 12 条复习产生 12 个 tab
3. **用户原有 tab 全部保留**：复习不破坏用户工作流
4. **每一步都是用户主动触发**：系统不自动推进

## 九、Cue 设计

### 正面展示内容

**标题 + TLDR**

### TLDR 提取优先级（fallback 链）

```
1. frontmatter 的 tldr 字段
   ↓ 没有就找
2. > [!tldr] callout
   ↓ 没有就找
3. ## TLDR 或 ## 摘要 区块下第一段
   ↓ 没有就用
4. 笔记的第一段（自然段落）
   ↓ 没有段落就用
5. 笔记开头前 N 字（N 可配置，默认 200）
```

### 设计原则

**用户任何写作习惯都能工作，但越用心写 TLDR，cue 质量越高**。这是一个"奖励好习惯但不惩罚坏习惯"的设计。

## 九·续、笔记编辑影响调度

### 核心设计

**所有编辑（不区分复习时/平时）在超过阈值时影响调度**。这是一条统一规则：
- 不区分复习状态还是平时编辑 → 心智模型简单，代码路径单一
- 只有"有意义的修改"才触发 → 避免改错别字也被当成大事件
- 符合产品世界观：**笔记是活的，它的变化会反过来影响调度**

### 阈值判断

**默认阈值**：改动字数 > 50 AND 改动比例 > 20%（两个条件都满足才触发）

- **绝对字数（默认 50 字）**：避免改错别字、调整标点等微小修改触发
- **相对比例（默认 20%）**：对长短笔记都公平
- **AND 取严**：防止误触发

### 触发后的动作

**默认动作**：将 Stability 乘以 0.5，然后让 FSRS 重新计算 next_review

- 最尊重 FSRS 算法（不直接改 next_review，而是改它的输入）
- 物理意义清楚：记忆"不那么稳定了"
- 温和：比"重置为新笔记"更缓和，比"乘 0.9"更有感觉

### 高级设置项

| # | 项 | 默认值 | 范围/选项 |
|---|---|---|---|
| 12 | 笔记编辑阈值 - 绝对字数 | 50 字 | 20-500 |
| 13 | 笔记编辑阈值 - 相对比例 | 20% | 5-50% |
| 14 | 超阈值时的动作 | Stability × 0.5 | × 0.3 / × 0.5 / × 0.7 / 重置为新笔记 / 不响应 |

### 工程实现要点

#### 快照存储

每条笔记入池后，额外存储：
- `lastSnapshotLength`：上次的总字符数
- `lastSnapshotHash`：上次的内容哈希

存长度和哈希而非全文，避免 data.json 膨胀。

#### 触发时机

- 监听 `vault.on('modify')`
- **Debounce 5 秒**：笔记停止编辑 5 秒后才做 diff 判断，避免打字过程中反复触发
- 触发 diff → 与 `lastSnapshot` 对比 → 超过阈值则执行触发动作 → 更新 snapshot

#### 防止无限放大

同一笔记短期内多次大改，避免 S 被反复压缩：
- **7 天内最多触发一次编辑影响**
- 触发后在 data 里记录 `lastEditTriggerAt`，7 天后才能再次触发

### 给想关闭该功能的用户

在"超阈值动作"里提供 "**不响应**" 选项。对于喜欢不断小修小补的用户（或者不希望编辑影响复习节奏的用户），可以彻底关闭此功能。

## 十、存储设计

### 存储位置

**Obsidian 插件 data.json**（通过 `saveData` / `loadData` API）

路径：`[vault]/.obsidian/plugins/obsidian-resurface/data.json`

### 数据结构（v0.1 草案）

```json
{
  "settings": {
    "dailyLimit": 15,
    "firstReviewDays": 3,
    "firstReviewJitter": 1,
    "ratingMode": "2-button",
    "desiredRetention": 0.9,
    "autoAdvance": false,
    "tldrFieldName": "tldr",
    "tldrFallbackLength": 200
  },
  "notes": {
    "<note-path.md>": {
      "stability": 7.3,
      "difficulty": 4.2,
      "lastReview": "2026-04-18T10:23:00Z",
      "nextReview": "2026-04-25T10:23:00Z",
      "state": "review",
      "excluded": false,
      "excludedAt": null,

      "lastSnapshotLength": 823,
      "lastSnapshotHash": "a1b2c3...",
      "lastEditTriggerAt": "2026-04-19T12:00:00Z"
    }
  },
  "revlog": [
    {
      "path": "<note-path.md>",
      "timestamp": "...",
      "rating": 3,
      "elapsedDays": 3,
      "stability": 7.3,
      "difficulty": 4.2
    }
  ],
  "stats": {
    "totalReviews": 435,
    "streakDays": 7,
    "lastReviewDate": "2026-04-21"
  }
}
```

### 同步策略

- Obsidian Sync 默认不同步 `.obsidian/plugins/`，需用户手动开启
- README 中说明同步约定
- MVP 不处理多设备冲突，未来版本可考虑 vault 内独立数据文件

### 笔记路径变更处理

必须监听两个 Obsidian 事件：
- `vault.on('rename')`：把 `data.notes[oldPath]` 搬到 `data.notes[newPath]`，同步更新 revlog 路径
- `vault.on('delete')`：从 data 里移除或归档

### 按钮视觉层级（刻意不对称）

同一界面里的两个按钮用不同视觉权重引导用户：

| 按钮 | 角色 | 视觉处理 |
|---|---|---|
| 展开正文 / 进入下一条 | 主按钮（95% 场景要点的那个） | accent 实心底 + 白字 + **font-weight 500** |
| 不再复习 | 次按钮（少数场景才用） | 透明底 + muted 文字色 + **font-weight 常规** |
| 评分「会」 | 主操作 | accent 底 + 白字 + 加粗 |
| 评分「不会」 | 次操作（心理上是"失败"） | 浅灰底 + muted 色 + 常规 |

**刻意不对称不是漏改**：用字重 + 色彩制造主次层级（primary/secondary hierarchy），引导用户自然走正常流程，降低"不再复习"这类半退出操作的误触风险。看起来不对称但心智上正确。

## 十二·续、文案风格指南

### 语气基调

- **克制、温暖、陈述事实**
- 避免：感叹号堆积、攀比数据、催促、鸡汤
- 允许：带轻微暗示意义感的文案（笔记是活的、会成长）

### 核心词汇

| 推荐 | 避免 |
|---|---|
| 重新见 / 唤醒 / 重新活一次 / 陪伴 | 学习 / 记忆 / 成就 / 进步 |
| "12 条笔记被重新见了一次" | "你复习了 12 条笔记" |
| "今天没有笔记到期" | "今日暂无待办" |

**原则**：笔记作为主语而不是"你"。产品不是学习工具，而是让笔记和你重逢的工具。

### 空状态文案示例

**今日无待复习**：
```
🌱 今天没有笔记等着被唤醒
下一批大约在 2 天后

连续 7 天陪伴笔记
累计复习 423 次
复活池里有 184 条笔记
```

**今日已完成**：

## 附录 A：术语表

- **FSRS**：Free Spaced Repetition Scheduler，开源间隔重复算法
- **DSR 模型**：Difficulty / Stability / Retrievability，FSRS 的记忆状态模型
- **R 值**：Retrievability，此刻回忆成功的概率
- **S 值**：Stability，记忆衰减到 R=90% 所需的天数
- **D 值**：Difficulty，卡片本身的难度
- **desired retention**：期望保留率，用户希望在复习时还能记得的概率
- **cue**：提示，复习时先看到的内容（标题 + TLDR）
- **TLDR**："too long; didn't read" 的摘要
- **revlog**：review log，完整复习历史记录
- **fuzz**：FSRS 间隔计算后的随机抖动，防止同批笔记同日到期
- **JOL**：Judgment of Learning，学习者对自己记住了多少的主观判断
- **streak**：连续复习天数
